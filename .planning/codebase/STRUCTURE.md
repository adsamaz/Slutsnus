## Directory Purposes

**Root Workspace (`/`):**
- Purpose: Monorepo orchestration with npm workspaces
- Contains: Shared scripts (dev, build, lint), root dependencies (Prisma, TypeScript, dev tools)
- Key files: `package.json` (workspaces array), `tsconfig.base.json` (base compiler options)

**Client Package (`client/`):**
- Purpose: Frontend SPA built with Solid.js and bundled by Vite
- Contains: TypeScript/TSX source, CSS, assets
- Key files: `src/index.tsx` (entry), `src/App.tsx` (router setup), `vite.config.ts` (build config)
- Output: `dist/` folder (built index.html, JS bundles)

**Server Package (`server/`):**
- Purpose: Express HTTP API + Socket.IO real-time server
- Contains: TypeScript source, database schema and migrations
- Key files: `src/index.ts` (Express setup), `prisma/schema.prisma` (database schema)
- Output: `dist/` folder (compiled JavaScript)

**Shared Package (`shared/`):**
- Purpose: Type definitions consumed by both client and server
- Contains: TypeScript interfaces (no implementation)
- Key files: `src/types.ts` (all interface definitions), `src/index.ts` (barrel export)
- Output: `dist/` folder with ESM and CJS builds

**Prisma Directory (`server/prisma/`):**
- Purpose: Database schema and migration management
- Contains: `schema.prisma` (data model), `migrations/` folder (SQL migration files)
- Generated: `.prisma/client/` folder (Prisma client code, excluded from git)

**Socket Handlers (`server/src/socket/`):**
- Purpose: Real-time event handling for rooms, games, friends
- Contains: Connection auth, event listeners, state sync broadcasts
- Usage: Imported and mounted in `server/src/index.ts` via initSocket()

**Game Engines (`server/src/games/`):**
- Purpose: Pluggable game implementations with standardized interface
- Contains: Abstract GameEngine interface, concrete implementations (SnusRpgEngine)
- Registry: Exported in `registry.ts` for dynamic instantiation

**Stores (`client/src/stores/`):**
- Purpose: Global state management using Solid.js context + providers
- Contains: Auth state, Socket connection, Room state, Friends list
- Pattern: Each store exports Provider component and useStore() hook

**Pages (`client/src/pages/`):**
- Purpose: Route-level components that compose stores and components
- Contains: Page logic, form handlers, navigation
- Pattern: Lazy-loaded in App.tsx router

**Components (`client/src/components/`):**
- Purpose: Reusable UI primitives
- Contains: Button, Modal, Toast (notifications), Navbar, ProtectedRoute guard
- Pattern: Exported individually, imported by pages/games

**Games (`client/src/games/`):**
- Purpose: Game-specific UI and rendering
- Contains: GameContainer wrapper, game implementations (snus-rpg with renderer, HUD, trade modal, end screen)
- Pattern: Receives state props from GamePage, emits socket actions via hooks

**Middleware:**

## Naming Conventions

**Files:**
- TypeScript files: `camelCase.ts` (e.g., `roomHandlers.ts`, `authMiddleware.ts`)
- TSX files: `PascalCase.tsx` (e.g., `GamePage.tsx`, `Lobby.tsx`, `Button.tsx`)
- Configuration: lowercase with extensions (e.g., `vite.config.ts`, `.prettierrc`)
- CSS files: Match component name in camelCase or PascalCase (e.g., `styles/global.css`, `snus-rpg/styles.css`)

**Functions:**
- Exported handlers: camelCase with suffix pattern (e.g., `roomHandlers`, `gameHandlers`, `authMiddleware`)
- Component functions: PascalCase (e.g., `function GamePage()`, `export default function Lobby()`)
- Hooks: camelCase with 'use' prefix (e.g., `useAuth()`, `useSocket()`, `useRoom()`)
- Utilities: camelCase (e.g., `buildRoomInfo()`, `signToken()`, `getState()`)

**Variables:**
- Constants: UPPER_SNAKE_CASE (e.g., `GAME_DURATION_TICKS`, `TICK_INTERVAL_MS`, `COOKIE_OPTIONS`)
- State variables: camelCase (e.g., `roomState`, `authState`, `tickCount`, `onlineUsers`)
- Socket event names: kebab-case (e.g., `'room:join'`, `'game:state'`, `'friends:invite'`)

**Types:**
- Interfaces: PascalCase without 'I' prefix (e.g., `RoomInfo`, `GameEngine`, `SnusRpgState`)
- Type aliases: PascalCase (e.g., `RoomStatus = 'waiting' | 'playing' | 'ended'`)
- Enums: PascalCase (not used in codebase, interfaces + unions preferred)
- Generics: Single uppercase letter (e.g., `<T>`) or descriptive PascalCase (e.g., `<ServerToClientEvents>`)

**Directories:**
- Feature folders: kebab-case or camelCase (e.g., `snus-rpg/`, `socket/`, `routes/`)
- Logical groups: lowercase plural or singular as appropriate (e.g., `pages/`, `components/`, `stores/`, `games/`)

## Where to Add New Code

**New Feature:**
- Primary code: `server/src/routes/[feature].ts` for HTTP endpoints, `server/src/socket/[feature].ts` for real-time events
- Types: Add interfaces to `shared/src/types.ts`
- Database: Update `server/prisma/schema.prisma`, run `prisma migrate dev`
- Client UI: Create page in `client/src/pages/[Feature].tsx`, add store if needed to `client/src/stores/[feature].tsx`

**New Game:**
- Implementation: Create `server/src/games/[game-name]/engine.ts` implementing GameEngine interface
- Registry: Register in `server/src/games/registry.ts` (gameRegistry object)
- Client renderer: Create `client/src/games/[game-name]/` with index.tsx, renderer, styles
- Shared types: Add game-specific types to `shared/src/types.ts`
- Test locally: Update room creation to use new gameType, test end-to-end via Socket.IO

**New UI Component:**
- Component file: `client/src/components/[ComponentName].tsx` (PascalCase)
- Styles: Co-locate CSS in same folder or reference from styles/ folder
- Pattern: Export as default, use Solid.js reactive primitives (createSignal, createEffect, etc.)
- Import in pages/games as needed

**Utilities or Helpers:**
- Shared helpers: `shared/src/utils.ts` (if logic is framework-agnostic)
- Server helpers: `server/src/utils.ts` or dedicated folder (e.g., `server/src/helpers/`)
- Client helpers: `client/src/utils/` folder with modules by concern
- Pattern: Export named functions, import where needed

**Database Operations:**
- All queries via Prisma client: Import from `server/src/db/client.ts`
- New models: Update `server/prisma/schema.prisma`
- Migrations: Run `prisma migrate dev --name [description]` in server directory
- Type safety: Prisma generates types automatically from schema
